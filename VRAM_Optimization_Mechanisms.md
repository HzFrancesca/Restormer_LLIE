# 深度学习模型训练显存优化机制与原理报告

在超分辨率或低光增强（如 Restormer）等高分辨率图像处理任务中，显存（VRAM）通常是训练性能的瓶颈。本报告深入探讨了两种核心显存优化技术：**梯度检查点 (Gradient Checkpointing)** 与 **梯度累积 (Gradient Accumulation)** 的设计原理及其在单卡环境下的应用。

---

## 1. 梯度检查点 (Gradient Checkpointing)

### 1.1 背景与痛点

在传统的 PyTorch 训练中，为了在反向传播（Backpropagation）时计算梯度，前向传播过程中每一层产生的激活值（Activations）都必须储存在显存中。

* 对于 Restormer 这种具有数十个 Transformer Block 的深度模型，激活值的显存占用随深度和分辨率成指数级增长。
* **后果**：即使 Batch Size 设为 1，在 384x384 甚至更高的分辨率下也会发生 Out of Memory (OOM)。

### 1.2 核心原理：空间换时间

梯度检查点通过“临时丢弃激活值”来节省空间。

* **前向传播 (Forward)**：模型正常计算，但仅在该 Block 的入口处保留一个“检查点”激活值。Block 内部转换产生的中间激活值全部被丢弃。
* **反向传播 (Backward)**：当梯度计算需要该 Block 内部的激活值时，程序会**重新执行一次前向传播**。利用入口处的备份值，再次计算出丢失的中间结果，随后完成梯度计算。

### 1.3 优缺点权衡

* **优势**：显存占用从“随深度线性增长”降低到“随深度的平方根增长”甚至更低。这使得在 24G 显存上运行大分辨率 Patch 成为可能。
* **代价**：由于每个模块在前向传播中计算了一次，在反向传播中为了找回数据又重算了一次，**计算量增加了约 33%**，导致训练速度下降。

## 2. 梯度累积 (Gradient Accumulation)

### 2.1 背景与痛点

深度学习需要较大的 Total Batch Size（总批大小）来保证梯度的稳定性和收敛质量。

* 原论文往往使用 8 张 32G 显卡（总 Batch=64）。
* 单卡环境下，如果不使用累积，物理 Batch Size 可能只能开到 4。小 Batch 会带来巨大的梯度噪声，导致模型不收敛或产生伪影。

### 2.2 核心原理：时间换空间

梯度累积本质上是一个“显存扩容术”。

* **逻辑流程**：
    1. 连续执行 $N$ 个物理平批次的前向传播。
    2. 每次计算出的梯度不立即更新权重，而是存储在 `param.grad` 中不断进行二进制累加。
    3. 只有当累加次数达到设定的 `Accumulation Steps` 时，才执行一次 `optimizer.step()` 更新权重。
* **数学等价性**：这种方式在数学上等价于在一个巨大的 Batch 上进行训练（除了 Batchnorm 等对统计量有要求的层会有极细微差异）。

### 2.3 效果

* **优势**：允许单卡模拟任意大规模集群的训练效果。
* **配置公式**：$\text{等价总 Batch} = \text{物理 Batch (显存限制)} \times \text{累积步数}$。

---

## 3. 协同工作逻辑：单卡强行对齐多卡方案

在我们的 Restormer 配置中，两者是如何配合的？

以 **GT 384 (Stage 6)** 阶段为例：

1. **梯度检查点 (开启)**：首先将原本无法运行的 384 patch 压缩进显存，允许物理 Batch 开到 **4**。
2. **梯度累积 (设为 8)**：连续跑 8 次物理 Batch。
3. **最终达成**：单张 4090 显卡实现了等效 **32** 的 Batch Size。

---

## 4. 优化效果对比总结

| 特性 | 梯度检查点 (Checkpointing) | 梯度累积 (Accumulation) |
| :--- | :--- | :--- |
| **解决目标** | 处理单个高分辨率图片导致的显存溢出 | 处理总 Batch Size 太小导致的收敛差 |
| **底层原理** | **重新计算**。丢弃激活值，反向传播再算回来 | **梯度叠加**。多次反向，一次更新 |
| **性能开销** | 增加约 30% 计算时间 | 增加参数更新的等待周期 |
| **对模型的影响** | 无任何数学精度损失 | 数学上等价于大 Batch 训练 |

---

## 5. 项目中的具体应用 (YAML 参考)

在 `Restormer_LOLv2_300k_1gpu.yml` 中：

```yaml
network_g:
  use_checkpoint: True  # 激活梯度检查点

train:
  mini_accumulation_steps: [1, 1, 2, 3, 4, 8] # 按阶段动态通过累积放大 Batch Size
```
